## The problem

The company built smart-building hardware and the software that ran on it, and had grown by shipping fast on instinct — which worked until it didn't. Mobile, desktop and hardware teams worked on largely separate timelines with no shared definition of "done," which meant integration surprises landed late, close to customer deadlines, when they were most expensive to fix.

## Constraints

- **Hardware doesn't iterate like software.** A firmware change or a sensor respin has lead times measured in weeks, not sprints — any process had to accommodate genuinely different cadences without pretending they were the same.
- **No prior PM function existed.** Introducing process to a team that had shipped successfully without one meant earning the right to add structure, not just importing a framework wholesale.
- **Small team, no slack.** There was no spare capacity to run parallel "how we work" experiments; changes had to work close to the first time.

## Decisions

Rather than impose one agile framework across all three teams, I set a single shared cadence for cross-team commitments — what would be integration-tested together and by when — while leaving each team's internal working style close to what already worked for them. Hardware kept longer internal cycles; software teams kept shorter ones. The shared layer was the interface between them, not their internal process.

I also introduced a lightweight definition-of-done that explicitly included cross-platform integration checks, which had previously been implicit and assumed rather than verified.

## Trade-offs

A fully unified agile process across all three disciplines would have been cleaner on paper and easier to explain, but would have forced hardware into a cadence it couldn't actually meet, or forced software to sit idle waiting on hardware's longer cycles. I chose the messier, hybrid model because it matched how the work actually happened, at the cost of a process that was harder to describe in one slide.

## Outcome

Cross-team integration surprises dropped noticeably, and the explicit definition-of-done caught mismatches during development instead of during customer pilots. The shared-cadence model became the default way new features were planned across the three platforms.

## What I learned

The instinct to standardize process across teams is usually about the process owner's convenience, not the teams' reality. The better question is which layer actually needs to be shared — often it's the interface between teams, not their internal workflow — and building process for a team with no prior PM function is as much about earning trust as it is about the framework itself.
